# Token Export Formats

> M7 reference. Three adapters that read `.design-master/design-system.json` and write to `.design-master/exports/`. All derived — no hand-maintenance.

The skill.md Step 4.7 describes *when* to export; this file is the format spec.

---

## Adapter pipeline

```
.design-master/design-system.json     (manifest, source of truth)
            │
            ├──► tokens.w3c.json              (canonical, W3C DTCG)
            │
            ├──► tokens.style-dictionary.json (Style Dictionary input)
            │
            └──► tokens.tokens-studio.json    (Tokens Studio for Figma)
```

W3C is canonical. The other two are light wrappers because both Style Dictionary v4+ and Tokens Studio natively accept W3C now — but each tool's documentation still leads with its native schema, so we ship that explicitly to reduce friction.

---

## 1. W3C DTCG (`tokens.w3c.json`)

Conforms to the [W3C Design Tokens Community Group format](https://design-tokens.github.io/community-group/format/). This is the canonical interchange format and the input to almost every modern tooling pipeline.

### Token-type mapping

CSS custom property → `$type`:

| Property pattern | `$type` |
|---|---|
| `--color-*` | `color` |
| `--font-display`, `--font-body`, `--font-mono` | `fontFamily` |
| `--font-weight-*` | `fontWeight` |
| `--font-size-*` | `dimension` |
| `--letter-spacing-*` | `dimension` |
| `--line-height-*` | `number` (unitless ratio) |
| `--radius-*` | `dimension` |
| `--shadow-*` | `shadow` |
| `--space-*`, `--grid-gap`, `--baseline` | `dimension` |
| `--motion-duration-*` | `duration` |
| `--motion-easing-*` | `cubicBezier` |

### Example output

```json
{
  "$schema": "https://design-tokens.github.io/community-group/format/",
  "$description": "Generated by AI Design Master (M7) from .design-master/design-system.json",

  "color": {
    "bg":     { "$value": "#ffffff", "$type": "color" },
    "ink":    { "$value": "#0A2540", "$type": "color", "$description": "brand-derived" },
    "accent": { "$value": "#635BFF", "$type": "color", "$description": "brand-derived" },
    "rule":   { "$value": "rgba(10,37,64,0.12)", "$type": "color" }
  },

  "font": {
    "display": {
      "$value": "Söhne Breit, Inter Tight, sans-serif",
      "$type": "fontFamily"
    },
    "body": {
      "$value": "Söhne, Inter Tight, sans-serif",
      "$type": "fontFamily"
    }
  },

  "weight": {
    "body":    { "$value": 400, "$type": "fontWeight" },
    "display": { "$value": 700, "$type": "fontWeight" }
  },

  "letter-spacing": {
    "display": { "$value": "-0.04em", "$type": "dimension" }
  },

  "radius": {
    "default": { "$value": "0", "$type": "dimension" }
  },

  "space": {
    "section": { "$value": "8rem", "$type": "dimension" }
  }
}
```

### Naming rules

- Strip the `--` prefix
- Replace the *first* hyphen with a group boundary: `--color-bg` → `color.bg`
- Multi-word categories keep internal hyphens: `--letter-spacing-display` → `letter-spacing.display`
- The output object is grouped by top-level category for clean tooling display

### Provenance preservation

Tokens with `tokens_provenance` value of `"brand"` get `$description: "brand-derived"`. Tokens with `"user-override"` get `$description: "user-override"`. Style-default tokens get no `$description` (default).

## 2. Style Dictionary (`tokens.style-dictionary.json`)

[Style Dictionary v4+](https://amzn.github.io/style-dictionary/) natively reads W3C DTCG. The "Style Dictionary input" we ship is **identical to the W3C output** with one addition: a `transforms` recommendation comment block at the top for users who want to compile to platform outputs (iOS, Android, web).

### Example output

Same JSON content as `tokens.w3c.json`, plus a leading comment in a separate `README-style-dictionary.md` file (since JSON has no comments):

```markdown
# Style Dictionary integration

This `tokens.style-dictionary.json` is W3C DTCG format. Style Dictionary v4+ reads it directly.

Quickstart:
```bash
npm install -D style-dictionary
npx style-dictionary build --source ".design-master/exports/tokens.style-dictionary.json" --config style-dictionary.config.js
```

Recommended `style-dictionary.config.js`:
```js
export default {
  source: ['.design-master/exports/tokens.style-dictionary.json'],
  platforms: {
    css: {
      transformGroup: 'css',
      buildPath: 'build/css/',
      files: [{ destination: 'tokens.css', format: 'css/variables' }]
    },
    js: {
      transformGroup: 'js',
      buildPath: 'build/js/',
      files: [{ destination: 'tokens.js', format: 'javascript/es6' }]
    }
  }
};
```
```

The skill writes both `tokens.style-dictionary.json` and `README-style-dictionary.md` so the user gets a working starter.

## 3. Tokens Studio for Figma (`tokens.tokens-studio.json`)

[Tokens Studio](https://tokens.studio/) is the dominant Figma plugin for design tokens. Its native schema is slightly different from W3C:

- Uses `value` (not `$value`), `type` (not `$type`)
- Wraps everything in a "set" (a Figma concept for token groupings)
- Stores extensions under `$extensions`

### Example output

```json
{
  "global": {
    "color": {
      "bg": {
        "value": "#ffffff",
        "type": "color"
      },
      "ink": {
        "value": "#0A2540",
        "type": "color",
        "description": "brand-derived",
        "$extensions": {
          "com.designsystem.provenance": "brand"
        }
      },
      "accent": {
        "value": "#635BFF",
        "type": "color",
        "description": "brand-derived",
        "$extensions": {
          "com.designsystem.provenance": "brand"
        }
      }
    },

    "font": {
      "display": {
        "value": "Söhne Breit, Inter Tight, sans-serif",
        "type": "fontFamilies"
      },
      "body": {
        "value": "Söhne, Inter Tight, sans-serif",
        "type": "fontFamilies"
      }
    },

    "weight": {
      "body":    { "value": "400", "type": "fontWeights" },
      "display": { "value": "700", "type": "fontWeights" }
    },

    "letter-spacing": {
      "display": { "value": "-0.04em", "type": "letterSpacing" }
    },

    "radius": {
      "default": { "value": "0", "type": "borderRadius" }
    },

    "space": {
      "section": { "value": "8rem", "type": "spacing" }
    }
  },

  "$themes": [],
  "$metadata": {
    "tokenSetOrder": ["global"]
  }
}
```

### Type mapping (Tokens Studio uses different keys)

| W3C `$type` | Tokens Studio `type` |
|---|---|
| `color` | `color` |
| `fontFamily` | `fontFamilies` |
| `fontWeight` | `fontWeights` |
| `dimension` (font-size) | `fontSizes` |
| `dimension` (radius) | `borderRadius` |
| `dimension` (space) | `spacing` |
| `dimension` (letter-spacing) | `letterSpacing` |
| `number` (line-height) | `lineHeights` |
| `shadow` | `boxShadow` |
| `duration` | `other` |
| `cubicBezier` | `other` |

The provenance is exposed in two places for visibility:
- `description: "brand-derived"` — visible in the plugin's standard UI
- `$extensions["com.designsystem.provenance"]` — readable by other tools / scripts

---

## Generation rules

1. **Always read from the manifest** — never re-derive tokens from the in-session state
2. **All three formats are regenerated in one call** — if the user runs "export tokens", produce all three (W3C + Style Dictionary + Tokens Studio) plus the Style Dictionary README. Cheap, and users may not know up front which one they need.
3. **Write to `.design-master/exports/`** — create the directory if absent
4. **Idempotent** — running export twice with the same manifest produces byte-identical output
5. **Round-trip safe** — a tool that reads the W3C output and emits CSS should produce the same `:root {}` block as the manifest

---

## Codex / no-MCP behavior

Token export is **pure file write**. No MCP / no rendering needed. Works identically on Codex, Claude Code, or any agent that can write files.

---

## What we deliberately don't export

- **Component CSS / HTML** — that's the *application* of tokens, not tokens themselves. Components are a separate export target (future work).
- **Per-page output** — Mode 7 already writes these to `.design-master/pages/`
- **Provenance to W3C `$extensions`** — W3C spec allows extensions but consumer support is uneven. Tokens Studio uses extensions as the well-known carrier; W3C uses `$description` for human-readable provenance. Two-channel by design.
- **`representativeSites`** / **`notAiTips`** — these are educational metadata in `data/design-bible.json`, not design tokens. They don't belong in a token export.
